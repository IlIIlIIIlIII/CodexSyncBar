# README UI 데모 캡처 설계

## 목표

README 상단에서 Codex SyncBar의 실제 사용자 UI를 바로 이해할 수 있도록 메인 메뉴와 설정 화면을 각각 한 장씩 제공한다. 캡처에는 실제 계정, 호스트, 사용량, 시각 또는 로컬 경로를 넣지 않는다.

## 선택한 방식

실제 `PopoverView`와 `SettingsView`를 그대로 렌더링하는 명시적 README 데모 모드를 추가한다. 기존 실사용 화면을 가리는 방식은 누락 위험이 있고, 별도 목업은 실제 앱과 쉽게 달라지므로 사용하지 않는다.

데모 모드는 일반 실행에서는 접근하지 않으며 캡처 명령에서만 활성화한다. 네트워크, Keychain, SSH, 인증 파일 및 사용자 설정을 읽거나 변경하지 않고 결정적인 가상 fixture만 표시한다.

## 산출물

- `docs/images/readme-popover.png`: 계정 선택, 사용량, 초기화권, 기기 요약이 보이는 메인 UI
- `docs/images/readme-settings.png`: 가상 계정 두 개와 순서 변경·계정 관리가 보이는 설정 UI
- README의 `사용자 UI` 섹션: 두 이미지를 설명과 함께 배치
- 반복 실행 가능한 캡처 명령 또는 스크립트

## 가상 데이터

- 계정: `demo.main@example.com`, `demo.sub@example.com`
- 별칭: `메인`, `서브`
- 장치: `이 MacBook`, `작업 서버`, `빌드 서버`
- 호스트: 문서용 예약 도메인인 `workstation.example.net`, `build.example.net`
- 사용량과 비용: 기능을 알아볼 수 있는 고정된 예시 값

실제 사용자 이메일, 사용자 이름, IP 주소, 홈 경로, SSH 호스트 이름은 fixture와 이미지에 포함하지 않는다.

## 캡처 구조

1. 캡처 전용 실행 인자가 들어오면 앱은 실사용 `AppModel` 시작 경로 대신 가상 fixture 모델을 사용한다.
2. 메인 또는 설정 SwiftUI 뷰를 고정 크기의 앱 소유 창에 렌더링한다.
3. 앱 자신의 content view를 직접 PNG로 저장하므로 화면 녹화 권한이나 다른 창의 픽셀에 의존하지 않는다.
4. 저장 후 앱은 자동으로 종료한다.

## 검증

- 전체 Swift 테스트와 helper 계약 테스트를 실행한다.
- 캡처 스크립트를 실행해 PNG 두 장이 생성되는지 확인한다.
- 두 PNG를 직접 열어 잘림, 스크롤, 빈 공간과 실제 UI 일치 여부를 확인한다.
- 저장소와 캡처 fixture에서 사용자의 실제 이메일, SSH IP, 사용자 이름 패턴이 존재하지 않는지 검색한다.
- README 이미지 링크가 저장소 상대 경로로 정상 해석되는지 확인한다.

## 범위 밖

- 앱의 일반 동작이나 데이터 모델 변경
- 실사용자 설정을 가상 데이터로 마이그레이션
- GIF 또는 동영상 제작
- 로그인 창과 실제 Google 인증 화면 캡처
